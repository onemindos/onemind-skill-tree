AWS Bedrock

Model-as-a-Service on Amazon Web Services

Overview

Amazon Bedrock is a fully managed service that makes high-performing foundation models from leading AI companies available through a single unified API. There is no infrastructure to provision, no model weights to store, and no GPUs to rent — AWS handles all of it. You pay only for the tokens you consume.

Bedrock integrates natively with the broader AWS ecosystem: IAM roles govern access instead of floating API keys, models run within your chosen AWS region for data-residency compliance, and traffic can traverse VPC private endpoints so tokens never cross the public internet. For organizations already inside AWS, Bedrock is often the fastest path to production AI.

Through Provisioned Throughput, teams can reserve model capacity for predictable high-volume workloads at rates that frequently beat per-token pricing on comparable direct-vendor plans.

Why Use Bedrock

🔑

IAM Authentication

No floating API keys in your codebase. Access is governed by IAM roles and policies — the same identity system that controls every other AWS service. Rotate credentials once, everywhere.

🌎

Regional Endpoints

Choose where your requests land: us-east-1, eu-west-1, ap-southeast-1, and more. Data sovereignty and GDPR compliance become architectural decisions, not afterthoughts.

🔒

VPC Private Endpoints

Route Bedrock traffic through an Interface VPC Endpoint. Tokens never traverse the public internet — essential for sensitive workloads, financial data, and healthcare (HIPAA-eligible).

📈

Cost via Commitments

On-demand pricing is competitive, but Provisioned Throughput unlocks reserved capacity at significant discounts for predictable workloads. Often cheaper than direct-vendor pricing at scale.

✅

Enterprise Compliance

SOC 2, ISO 27001, PCI DSS, FedRAMP (select regions), and HIPAA BAA available. Bedrock logs all model invocations to CloudTrail for full audit trails out of the box.

👥

Single API Surface

One SDK (boto3), one authentication mechanism, one monitoring stack (CloudWatch) — and access to Claude, Llama, Mistral, Titan, Cohere, and more through the same interface.

How to Connect

Step 1 — AWS CLI Setup

Configure credentials using the AWS CLI. This writes to ~/.aws/credentials and ~/.aws/config. 

# Install AWS CLI then run:
aws configure

# You will be prompted for:
# AWS Access Key ID      [None]: AKIAIOSFODNN7EXAMPLE
# AWS Secret Access Key  [None]: wJalrXUtnFEMI/K7MDENG/...
# Default region name    [None]: us-east-1
# Default output format  [None]: json

Step 2 — Environment Variables

Alternatively, set env vars directly. These override ~/.aws/credentials at runtime. 

export AWS_ACCESS_KEY_ID="AKIAIOSFODNN7EXAMPLE"
export AWS_SECRET_ACCESS_KEY="wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY"
export AWS_DEFAULT_REGION="us-east-1"

# For IAM roles on EC2/ECS/Lambda — no env vars needed at all.
# boto3 picks up the instance metadata role automatically.

Step 3 — API Endpoint Pattern

Bedrock Runtime uses a region-scoped endpoint. All model inference goes through this base: 

# Runtime endpoint (inference):
https://bedrock-runtime.{region}.amazonaws.com

# Example (Claude Sonnet 4, us-east-1):
https://bedrock-runtime.us-east-1.amazonaws.com/model/anthropic.claude-sonnet-4-20250514-v1:0/invoke

# Control plane (list models, manage throughput):
https://bedrock.{region}.amazonaws.com

Step 4 — Python (boto3)

Use the bedrock-runtime client. The API is identical regardless of which model you target. 

import boto3, json

client = boto3.client(
    "bedrock-runtime",
    region_name="us-east-1"
)

response = client.invoke_model(
    modelId="anthropic.claude-sonnet-4-20250514-v1:0",
    contentType="application/json",
    accept="application/json",
    body=json.dumps({
        "anthropic_version": "bedrock-2023-05-31",
        "max_tokens": 1024,
        "messages": [{"role": "user", "content": "Hello, Claude!"}]
    })
)

result = json.loads(response["body"].read())
print(result["content"][0]["text"])
⚠ You must enable model access in the AWS Bedrock console before you can invoke any model. Navigate to Bedrock → Model access → Manage model access → check each model → Save changes. Approval is usually instant but can take up to a few minutes.

Used By

⌨

Claude Code

Coding Agent — via Bedrock profile

Claude Code's --bedrock flag (or the CLAUDE_CODE_USE_BEDROCK env var) routes all model calls through Bedrock Runtime instead of api.anthropic.com. Ideal for teams that already operate inside AWS and want unified billing + IAM governance.

☁

Hermes Agent

Autonomous Orchestrator — fallback provider

Hermes Agent is configured with AWS Bedrock as a fallback provider. When the primary provider is unavailable or rate-limited, Hermes routes inference through Bedrock using the same Claude models, ensuring continuity for long-running agentic workflows.

🛸

OpenClaw

Multi-Agent Fleet — cloud inference backend

OpenClaw's multi-agent orchestration layer uses Bedrock as one of its cloud inference backends, enabling cost-optimized fleet scaling: route cheaper subtasks to Haiku, complex reasoning to Sonnet or Opus, all under a single AWS bill.
